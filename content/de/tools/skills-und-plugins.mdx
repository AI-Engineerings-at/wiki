## SKILL.md Frontmatter вҖ” Alle Felder

| Feld | Typ | Default | Beschreibung | Beispiel |

## String Substitutionen вҖ” Dynamische Werte

Skills unterstГјtzen String-Substitution fГјr dynamische Werte im Skill-Content:

| Variable | Beschreibung | Beispiel |
|----------|-------------|---------|

### Substitutions-Beispiele

**Mit mehreren Argumenten (indexed):**

```yaml

description: Migriert eine Komponente
---

Migriere $0 von $1 zu $2. Behalte Verhalten und Tests bei.
```

Aufruf: `/migrate-component SearchBar React Vue`
вҶ’ $0=SearchBar, $1=React, $2=Vue

**Mit Session-Logging:**

```yaml

---

```

**Mit Skill-Verzeichnis:**

```yaml

allowed-tools: "Bash(python *)"
---

```

WICHTIG: `${CLAUDE_SKILL_DIR}` verwenden statt hardcoded Pfade!

## Dynamische Kontext-Injektion вҖ” Live-Daten

Die `` !`<command>` `` Syntax fГјhrt Shell-Commands AUS BEVOR der Skill an Claude gesendet wird. Das Output ersetzt den Placeholder:

```yaml

---

- PR diff: !`gh pr diff`

- Changed files: !`gh pr diff --name-only`

```

**Ablauf:**
1. Jeder `` !`<command>` `` wird SOFORT ausgefГјhrt
2. Das Output ersetzt den Placeholder
3. Claude erhГӨlt die vollstГӨndig gerenderte Prompt mit echten Daten

Das ist Preprocessing, nicht etwas das Claude ausfГјhrt!

## Skill Locations вҖ” Scope und Priority

Wo du einen Skill speicherst, bestimmt Scope und Priority:

| Location | Pfad | Anwendbar auf | Priority |
|----------|------|--------------|----------|
| Enterprise | (Managed Settings) | Alle Org-User | 1 (hГ¶chste) |
| Personal | `~/.claude/skills/<name>/SKILL.md` | Alle deine Projekte | 2 |
| Project | `.claude/skills/<name>/SKILL.md` | Dieses Projekt nur | 3 |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | Wo Plugin enabled | 4 (niedrigste) |

**Priority-Regel:** Enterprise > Personal > Project > Plugin.

**Automatic Nested Discovery:** Claude Code findet Skills automatisch in nested `.claude/skills/` Verzeichnissen. In Monorepos: `packages/frontend/.claude/skills/`.

## Wer darf Skills aufrufen?

### `disable-model-invocation: true` вҖ” Nur User aufrufen

Verhindert dass Claude automatisch lГӨdt. Nutzen fГјr Workflows mit Seiteneffekten:


- Du kannst aufrufen вқҢ
- Claude ruft automatisch auf вң…
- In Claude's Context вң…

|-------------|------|--------|---------|

## Tool-Zugriff limitieren вҖ” `allowed-tools`

Mit `allowed-tools` kannst du Tools restricten:

```yaml
---

description: Liest Dateien ohne Г„nderungen

---
```

**Best Practice:** Principle of Least Privilege:
- Read-Only: `Read, Grep, Glob`
- Content Creation: `Read, Grep, Glob, Write, Edit`
- Deployment: `Read, Grep, Glob, Bash`

## Hooks in Skills вҖ” Lifecycle-Automation

Skills kГ¶nnen Hooks definieren fГјr bestimmte Events:

```yaml
---
name: code-reviewer
description: Reviews Code mit automatischem Linting
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-command.sh"
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "./scripts/run-linter.sh"
---
```

**Hook-Events:**
- `PreToolUse`: Bevor Claude ein Tool nutzt (Validation)
- `PostToolUse`: Nachdem ein Tool genutzt wurde (Cleanup)
- `Stop`: Wenn Skill beendet wird

Hook-Scripts bekommen Input via stdin. Exit Code 0 (OK) oder 2 (Block).

## Skill Discovery вҖ” Wann nutzt Claude Skills?

Claude kennt eure Skills durch ihre `description`. Je besser die Description, desto besser die Auto-Detection:

- **Gut:** "ErklГӨrt Code mit Diagrammen. Nutzen wenn erklГӨrt wird wie etwas funktioniert."
- **Schlecht:** "Code-Tool" (zu vage)

Claude's Skill Context hat ein Budget (2% des Context Windows, min 16KB). `/context` zeigt ob Descriptions excluded wurden. `SLASH_COMMAND_TOOL_CHAR_BUDGET` Env-Var kann Budget setzen.
